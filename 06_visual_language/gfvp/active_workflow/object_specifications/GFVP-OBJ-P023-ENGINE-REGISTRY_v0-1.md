## Purpose

Engine Registry records named engines, their allowed uses, constraints, statuses, and review triggers.

## Constitutional Role

It matters because engines must remain visible and governed rather than becoming invisible infrastructure with assumed authority.

## Boundaries

Engine Registry is distinct from Engine because it lists and governs engines but does not execute transformation. It is distinct from Product Boundary because it is an institutional registry, not a platform feature.

## Inputs / Outputs

### Inputs

- engine name
- allowed use
- constraint note
- status update
- review trigger

### Outputs

- registry entry
- allowed-use statement
- constraint record
- engine status
- review queue

### Non-Outputs

- engine execution
- approval
- canon
- automation authority
- product dashboard

## Required Labels

- Engine Registry
- Engine Name
- Allowed Use
- Constraint Record
- Engine Status
- Registry is not execution

## Forbidden Concepts

- canon by default
- visual approval as authority
- automatic promotion or ratification
- software interface or dashboard
- production workflow
- prompt engineering
- QC process as subject matter
- image gate as subject matter
- conversation summary
- mystical seal or certification badge
- object collapse into adjacent roadmap plates
