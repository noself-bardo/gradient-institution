## Purpose

Capability names a human or institutional capacity that can be preserved, supported, extended, or found missing.

## Constitutional Role

It matters because The Gradient must protect capabilities as capacities, not confuse them with tools, people, engines, or outputs.

## Boundaries

Capability is distinct from Capability Map because it names one capacity, while the map relates many capacities. It is distinct from Engine because an engine may support a capability but is not the capability.

## Inputs / Outputs

### Inputs

- capacity name
- human substrate
- support condition
- use context
- limit note

### Outputs

- capability statement
- support path
- preservation status
- limit marker
- review need

### Non-Outputs

- person replacement
- product feature
- engine output
- guarantee
- canon

## Required Labels

- Capability
- Human Substrate
- Support Condition
- Preservation Status
- Limit Marker
- Capability is not replacement

## Forbidden Concepts

- canon by default
- visual approval as authority
- automatic promotion or ratification
- software interface or dashboard
- production workflow
- prompt engineering
- QC process as subject matter
- image gate as subject matter
- conversation summary
- mystical seal or certification badge
- object collapse into adjacent roadmap plates

## Orientation

Landscape or near-square is acceptable when it improves legibility of the named structure.

No skill badge, no HR competency chart, no superhero icon, no product feature card, no automation replacement, no performance score. No paper mockup, no shadows, no perspective, no UI screen, no dashboard, no app card, no corporate infographic, no colorful categories, no neon, no random glyphs, no prompt engineering, no QC document, no image-generation process, no image gate, no conversation summary, no automatic canon language, no final authority language, no invented metadata.

## Compiler Notes

Render only Capability. Keep adjacent objects subordinate or absent unless listed as labels. Use GFVP field atlas grammar with matte black background, metallic silver and pale institutional gray typography and linework only, strict modular grid, high negative space, thin functional rules, and Sandbox / Candidate status language.

- inputs and outputs match the object-specific lists above
