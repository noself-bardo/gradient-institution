## Purpose

Product / Platform Boundary keeps tools, interfaces, and software platforms subordinate to institutional design.

## Constitutional Role

It matters because convenience, interface defaults, or platform capability must not become governance.

## Boundaries

It is distinct from Automation Boundary because it concerns the product/platform servant layer broadly, while Automation Boundary concerns automated authority. It is distinct from Commons because it governs tool relation, not shared knowledge access.

## Inputs / Outputs

### Inputs

- institutional need
- tool capability
- platform constraint
- steward boundary
- use case

### Outputs

- servant-layer role
- use limit
- boundary reminder
- escalation note
- non-governance signal

### Non-Outputs

- product authority
- platform governance
- canon
- approval
- constitutional decision

## Required Labels

- Product / Platform Boundary
- Institutional Need
- Servant Layer
- Use Limit
- Boundary Reminder
- Platform does not govern

## Forbidden Concepts

- canon by default
- visual approval as authority
- automatic promotion or ratification
- software interface or dashboard
- production workflow
- prompt engineering
- QC process as subject matter
- image gate as subject matter
- conversation summary
- mystical seal or certification badge
- object collapse into adjacent roadmap plates
