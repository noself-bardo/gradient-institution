## Purpose

Validation Gate explains the institutional threshold where a candidate object is tested against practice, evidence, criteria, and steward review.

It determines whether the object works well enough to proceed, revise, hold, or reject.

## Constitutional Role

Validation protects The Gradient from adopting attractive but untested structures.

It makes working evidence more important than aesthetic coherence, enthusiasm, or conversational momentum.

It preserves the distinction between candidate usefulness and institutional authority.

## Boundaries

Validation Gate tests whether an object works in practice.

It is not Admission.

It is not Source Review.

It is not Metabolization.

It is not Promotion.

It is not Canon.

It is not final truth.

It does not make an object constitutional.

It does not approve an image by itself.

It may send an object forward, backward, sideways, or out.

## Diagram Family

Gate / Test Plate.

Primary structure: one central Validation Gate with the Candidate Object entering and four visible dispositions: Validated for Use, Revise and Retest, Hold for Evidence, Reject.

Promotion review may appear only as a subordinate next-review output after validation.

Do not render a software testing dashboard, grading sheet, certification badge, or corporate KPI chart.

## Visual Center

One severe central object labeled `VALIDATION GATE`.

The gate should show criteria and practice evidence as the testing field.

Disposition exits must remain visible and distinct.

The visual hierarchy must make validation a test threshold, not a final authority.

## Orientation

Landscape is acceptable and likely appropriate for gate crossing.

Portrait or near-square is acceptable if the disposition exits remain readable.

The orientation reason is threshold testing, not a software workflow.
