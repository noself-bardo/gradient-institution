## Purpose

Capability Map relates capabilities to supports, engines, practices, statuses, and gaps.

## Constitutional Role

It matters because the institution needs to see where capacities are supported, missing, overclaimed, or dependent.

## Boundaries

Capability Map is distinct from Capability because it is relational cartography, not one capacity. It is distinct from Preserved Human Capability Matrix because it maps relationships broadly rather than focusing on preservation status.

## Inputs / Outputs

### Inputs

- capability record
- support object
- engine relation
- practice relation
- gap marker

### Outputs

- mapped relation
- support cluster
- dependency note
- gap marker
- review priority

### Non-Outputs

- new capability
- approval
- product roadmap
- staffing chart
- canon

## Required Labels

- Capability Map
- Capability Record
- Support Object
- Dependency Note
- Gap Marker
- Map is not capability

## Forbidden Concepts

- canon by default
- visual approval as authority
- automatic promotion or ratification
- software interface or dashboard
- production workflow
- prompt engineering
- QC process as subject matter
- image gate as subject matter
- conversation summary
- mystical seal or certification badge
- object collapse into adjacent roadmap plates

## Diagram Family

Matrix / Cartography Plate.

Structure: A cartographic matrix linking capability nodes to support objects, dependency notes, and gap markers.

## Visual Center

A cartographic matrix linking capability nodes to support objects, dependency notes, and gap markers.

## Orientation

Landscape or near-square is acceptable when it improves legibility of the named structure.

## Negative Prompt

No geographic map, no org chart, no product roadmap, no talent matrix, no colorful quadrant chart, no capability scorecard. No paper mockup, no shadows, no perspective, no UI screen, no dashboard, no app card, no corporate infographic, no colorful categories, no neon, no random glyphs, no prompt engineering, no QC document, no image-generation process, no image gate, no conversation summary, no automatic canon language, no final authority language, no invented metadata.

## Compiler Notes

Render only Capability Map. Keep adjacent objects subordinate or absent unless listed as labels. Use GFVP field atlas grammar with matte black background, metallic silver and pale institutional gray typography and linework only, strict modular grid, high negative space, thin functional rules, and Sandbox / Candidate status language.

## QC Requirements

Pass only if:

- Capability Map remains the single object depicted
- boundaries distinguish it from adjacent roadmap objects
- inputs and outputs match the object-specific lists above
- non-outputs are not implied as outputs
- labels are exact and object-specific
- diagram structure matches the described visual center
- the negative prompt blocks the named drift risks
