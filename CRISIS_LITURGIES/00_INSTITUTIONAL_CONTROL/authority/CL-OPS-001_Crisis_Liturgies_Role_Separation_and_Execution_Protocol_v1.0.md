# CL-OPS-001 — Crisis Liturgies Role Separation and Execution Protocol v1.0

**Status: LOCKED / EFFECTIVE**  
**Ratified: July 12, 2026**  
**Authority: Founder ratification; subordinate to Crisis Liturgies Canon v1.1, CCR-CL-002, and CL-GEN-STD-001.**

# Decision

Crisis Liturgies production will operate through separated authority and execution roles. No single model, application, chat, repository, or handoff may simultaneously own canon interpretation, reconstruction design, builder implementation, local execution, quality judgment, and release admission.

# Governing principle

Codex builds the machine. Cursor operates the machine. The Conversion Architect defines what is institutionally and aesthetically valid. Chachipti compiles bounded corrective instructions. The Founder ratifies canon and release decisions.

# Authority hierarchy

1. Founder-ratified canon and accepted Canon Change Reports  
2. Crisis Liturgies Generation Standard  
3. Approved release profile  
4. Frozen volume specification  
5. Approved Conversion Architecture and Render Contract  
6. Page specification  
7. Chachipti execution packet  
8. Builder implementation  
9. Renderer or model output

## Role 1 — CL-CONV-001 / Conversion Architect

Owns intake, chain of custody, lineage reconstruction, failure autopsy, source arbitration, canon recovery, reconstruction architecture, page rebuild maps, guardrails, render contracts, and post-render institutional and aesthetic QC.  
May not render without authorization, alter ratified canon, overwrite source artifacts, or treat technical completion as aesthetic completion.

## Role 2 — CL-CHACHIPTI-001 / Corrective Compiler

Owns conversion of an approved reconstruction plan into narrow Cursor and Codex packets; preservation of prior releases; failure-class encoding; exact path, scope, retry, stop-condition, and evidence-return requirements.  
It is not a constitution, canon authority, replacement for CL-CONV-001, replacement for the builder, release authority, or autonomous volume designer.

## Role 3 — Codex / Builder Engineer

Owns schemas, validators, compilers, build scripts, automated QC, repair routing, PDF assembly, checksums, manifests, regression tests, build locks, and reproducible release candidates.  
May not decide canon, rewrite approved material, infer missing approval, or promote a technical pass into a final Crisis Liturgies pass.

## Role 4 — Cursor / Repository Operator

Owns repository inspection, scoped edits, local command execution, path repair, candidate generation, diffs, logs, and bounded corrective reruns.  
May not redefine Crisis Liturgies, universalize volume-specific rules, promote local status into institutional status, modify protected files, invent missing canon, overwrite frozen releases, or expand scope.

## Role 5 — Renderer

Receives one approved render packet, executes it, and returns output plus technical metadata. It may not consult archive history, write authoritative publication copy, choose page functions, or exercise editorial authority.

## Role 6 — Founder

Owns canon ratification, concept approval, APPROVED\_FOR\_RENDER, unresolved aesthetic judgment, exception admission, and final release acceptance or rejection.

# Authorization

Builder authorization follows CCR-CL-002:

* CONCEPT\_APPROVED  
* APPROVED\_FOR\_RENDER

After APPROVED\_FOR\_RENDER is recorded in a frozen specification, Codex and Cursor may execute approved rendering, repair, assembly, and validation without repeated conversational authorization.  
Any native ChatGPT image-generation trigger convention remains separate from builder state and does not govern an approved Cursor or Codex build.

# Required execution sequence

1. CL-CONV-001 receives the volume or failed artifact.  
2. CL-CONV-001 establishes chain of custody and lineage.  
3. CL-CONV-001 issues the failure autopsy and Canon Recovery Map.  
4. CL-CONV-001 issues the Reconstruction Architecture, Page Rebuild Map, Guardrails, and Render Contract.  
5. Chachipti compiles bounded Cursor and Codex packets.  
6. Codex validates or implements deterministic machinery.  
7. Cursor executes locally and returns diffs, logs, manifests, checksums, and exceptions.  
8. Codex runs objective automated QC.  
9. CL-CONV-001 runs institutional, editorial, and visual QC.  
10. The Founder approves, rejects, or returns the release candidate.

# Stop conditions

* Two authoritative sources conflict.  
* Required source files are missing.  
* A protected file would need modification.  
* Canon judgment is required.  
* The retry ceiling is reached.  
* An approved requirement is technically impossible.  
* Current state cannot be verified.  
* The candidate cannot be traced to the frozen specification.

# Final acceptance rule

A technically valid build is not automatically a finished Crisis Liturgies volume. Final acceptance requires technical compliance, correct page functions, visual-system coherence, editorial integration, civilizational distance, object-as-witness integrity, complete provenance, and Founder acceptance.
